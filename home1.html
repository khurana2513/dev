01

Core Concepts — Understand This First

Before wiring anything, you need to understand how electricity flows in your room and why relays exist. Let's use a simple analogy.

The Water Analogy

⚡

Live Wire (Phase)

Like a pipe with pressurized water — it carries the actual electricity. In India: 230V AC, Red/Brown wire. This is dangerous, always respect it.

🔵

Neutral Wire

Like the drain pipe — completes the circuit. Current flows Live → Load → Neutral. Black/Blue wire. Also present at high current levels.

🟢

Earth/Ground Wire

Safety wire — only carries current if something goes wrong. Green/Yellow wire. Never skip this in professional installs.

🔌

Load

The device you're controlling — a bulb, fan, socket. It sits between Live and Neutral in the circuit. The switch breaks or completes this path.

How a Switch Actually Works

// Basic Switch Circuit
  LIVE (230V) ──────┬──── [SWITCH] ────── [BULB] ──────┐
                    │                                      │
                    │    When switch OPEN  → No current   │
                    │    When switch CLOSE → Bulb lights  │
                    │                                      │
  NEUTRAL ──────────────────────────────────────────────┘

  Simple, right? A relay is just an ELECTRICALLY CONTROLLED switch.
  Instead of your finger pressing it, a 5V signal from NodeMCU triggers it.
💡
Key Insight:

Your existing wall switches are just mechanical switches interrupting the Live wire. Relays replace (or sit in parallel with) these switches, letting NodeMCU do the same thing electronically.

02

Relay Deep Dive — NO vs NC Explained

This is where most beginners get confused. A relay has two sides — a control side (low voltage) and a switch side (high voltage). They are electrically isolated.

Relay Pin Layout

  ┌─────────────────────────────────────────────────┐
  │                  RELAY MODULE                    │
  │                                                  │
  │  CONTROL SIDE (5V)    │   SWITCH SIDE (230V AC)  │
  │  ┌──────────────┐     │   ┌──────────────────┐   │
  │  │ VCC  GND  IN │     │   │ COM   NO    NC   │   │
  │  └──────────────┘     │   └──────────────────┘   │
  │   ↑     ↑     ↑       │     ↑      ↑      ↑      │
  │  +5V   GND  GPIO      │  Common  N.Open  N.Closed │
  │                       │                           │
  │  NodeMCU controls ────┘── This side controls 230V │
  └─────────────────────────────────────────────────┘

  COM = Common — always connect your Live wire here
  NO  = Normally Open — circuit is OPEN when relay is OFF
  NC  = Normally Closed — circuit is CLOSED when relay is OFF

NO (Normally Open) — Most Common for Lights

DEFAULT CHOICE

Normally Open (NO)

Circuit is broken (open) by default. Relay activates → circuit closes → device turns ON.

  Relay OFF (default):   COM ──╳── NO   (circuit broken, device OFF)
  Relay ON (energized):  COM ─────── NO  (circuit complete, device ON)

  Wiring: LIVE → COM → NO → [BULB] → NEUTRAL

  NodeMCU GPIO HIGH → Relay triggers → NO connects to COM → Bulb ON
  NodeMCU GPIO LOW  → Relay resets   → NO disconnects    → Bulb OFF
✓ Advantages
  • Safe default — device is OFF if NodeMCU loses power or crashes
  • Standard for lights and fans
  • Less wear on relay contacts
  • Lower power consumption (relay coil not energized at rest)
✗ Disadvantages
  • Devices turn OFF on NodeMCU reset/reboot
  • Not ideal for things that must stay ON during power outages

NC (Normally Closed) — For Fail-Safe Applications

SPECIAL USE CASE

Normally Closed (NC)

Circuit is complete (closed) by default. Relay activates → circuit opens → device turns OFF.

  Relay OFF (default):   COM ─────── NC  (circuit complete, device ON)
  Relay ON (energized):  COM ──╳── NC   (circuit broken, device OFF)

  Wiring: LIVE → COM → NC → [DEVICE] → NEUTRAL

  NodeMCU GPIO HIGH → Relay triggers → NC disconnects → Device OFF
  NodeMCU GPIO LOW  → Relay resets   → NC connects    → Device ON

  LOGIC IS INVERTED — LOW turns device ON, HIGH turns it OFF
⚠️
When to use NC:

Use NC for devices that should remain ON if your NodeMCU dies — like security lights, refrigerators, or heaters in cold climates. For normal home automation (lights, fans), always use NO.

Important: Active LOW vs Active HIGH Relays

⚠️
Most relay modules are ACTIVE LOW!

This means sending GPIO LOW (0V) = Relay ON, and GPIO HIGH (3.3V) = Relay OFF. This is counterintuitive but very common. Always check your module. In code, initialize pins HIGH to keep relays off on boot, and write LOW to turn them on.

Active LOW Relay Logic
// For ACTIVE LOW relay modules (most common Chinese modules)
pinMode(RELAY_PIN, OUTPUT);
digitalWrite(RELAY_PIN, HIGH);  // Initialize OFF (relay not energized)

// To turn device ON:
digitalWrite(RELAY_PIN, LOW);   // Activates relay → device ON

// To turn device OFF:
digitalWrite(RELAY_PIN, HIGH);  // Deactivates relay → device OFF

Wiring Methods Compared — All Your Options

You mentioned finding multiple wiring methods. Here's every approach, explained with its trade-offs.

Method A: Direct Relay Replacement (No Manual Switch)

NOT RECOMMENDED

Remove wall switch, relay-only control

Relay completely replaces the manual switch. IoT-only control.

  LIVE ──── [RELAY COM] ──── [RELAY NO] ──── [BULB] ──── NEUTRAL

  Wall switch is completely removed or bypassed.
  Only NodeMCU can control the device.
✓ Pros
  • Simple wiring
  • Clean circuit
✗ Cons
  • No manual control if WiFi/NodeMCU fails
  • Guests can't use the switch
  • Not practical at all

Method B: Relay in Series with Switch (Both control independently)

AVOID

Switch AND relay both in series

BOTH must be ON for device to work. If switch is OFF, relay can't turn it on.

  LIVE ─── [WALL SWITCH] ─── [RELAY COM→NO] ─── [BULB] ─── NEUTRAL

  Device ON only when: Switch=ON AND Relay=ON
  This is a logic AND gate — almost never what you want.
✓ Pros
  • Has a "master off" via wall switch
✗ Cons
  • Must leave switch ON at all times for IoT to work
  • Confusing for users
  • Not independent control

Method C: Relay in Parallel with Switch ⭐ BEST METHOD

✓ RECOMMENDED

Relay bypasses the existing wall switch

The relay is wired in PARALLEL with the existing wall switch. Either the switch OR the relay can independently turn the device on. This is the professional standard.

                    ┌──── [WALL SWITCH] ────┐
  LIVE ─────────────┤                       ├──────── [BULB] ──── NEUTRAL
                    └──── [RELAY COM→NO] ────┘

  Think of it like two doors into the same room:
  - Either door being open = you can enter (device ON)
  - Wall switch ON → device ON  (even if relay is off)
  - Relay ON → device ON        (even if switch is off)
  - Both ON → device ON         (no problem, they're parallel)

  TRUTH TABLE:
  Switch=OFF, Relay=OFF → Device OFF
  Switch=ON,  Relay=OFF → Device ON  (manual control works)
  Switch=OFF, Relay=ON  → Device ON  (IoT control works)
  Switch=ON,  Relay=ON  → Device ON  (both ON = still just ON)
🚨
State Sync Problem — You MUST handle this!

If the light is ON via switch (relay=OFF), and you send an IoT "turn OFF" command, the relay turns off — but the switch is still ON, so the light stays on! Solution: Use smart switches that can detect their position (like momentary push buttons), or use the relay to control power to the switch circuit itself. See Section 5 for the code solution.

✓ Pros
  • Full manual control preserved
  • Full IoT control
  • Professional standard approach
  • Safe fallback if WiFi dies
  • Guests can use switches normally
✗ Cons
  • State synchronization needs handling in code
  • Slightly more complex wiring

Method D: Replace Switches with Momentary Push Buttons (Premium)

PREMIUM / PROFESSIONAL

Momentary buttons + relay = perfect sync

Replace toggle switches with momentary push buttons. Each button press TOGGLES the relay state via NodeMCU. This completely solves the state sync problem and is used in all professional smart home products.

  [PUSH BUTTON] ──── GPIO Input (interrupt-driven)
                           │
                           ▼
                    NodeMCU toggles relay
                           │
                           ▼
                    [RELAY] ──── [BULB]

  Press button → NodeMCU sees falling edge → Toggles relay state
  Relay tracks state perfectly — IoT and button always in sync!
  This is how Sonoff, Shelly, and other smart switches work internally.
✓ Pros
  • Perfect state synchronization
  • No confusion between switch and relay state
  • Debounce easily handled in code
  • Industry standard for smart switches
✗ Cons
  • Must replace existing toggle switches
  • Requires electrician if not DIY
  • Push buttons feel different from traditional switches

4-Board Architecture — How to Connect Everything

With 4 switchboards, you have multiple devices to control. The key decisions: centralized vs distributed NodeMCU, and how many relays per board.

Architecture Option A: One Central NodeMCU (Best for your case)

RECOMMENDED FOR 4 BOARDS

Single NodeMCU + 8-channel relay module controls all boards

  ┌─────────────────────────────────────────────────────────────┐
  │                    CENTRAL CONTROL BOX                       │
  │                                                              │
  │   [NodeMCU ESP8266]  ←→  WiFi / Internet                    │
  │        │                                                     │
  │        └── [8-Channel Relay Module]                          │
  │               │                                              │
  │               ├── Relay 1 ── Board 1, Light 1                │
  │               ├── Relay 2 ── Board 1, Fan                    │
  │               ├── Relay 3 ── Board 2, Light 1                │
  │               ├── Relay 4 ── Board 2, Light 2                │
  │               ├── Relay 5 ── Board 3, Light                  │
  │               ├── Relay 6 ── Board 3, Socket                 │
  │               ├── Relay 7 ── Board 4, Light                  │
  │               └── Relay 8 ── Board 4, Fan                    │
  │                                                              │
  └─────────────────────────────────────────────────────────────┘

  Manual switches on each board connect to NodeMCU GPIO inputs
  (with 10kΩ pull-up resistors) for state tracking
🔧

8-Channel Relay Module

One board with 8 isolated relays. Controls up to 8 devices. ~₹200-400. Uses opto-isolation for safety. Get the blue Songle brand modules.

📦

4-Channel Relay Module

If you have only 4 devices total (one per board). Simpler, cheaper. ~₹150-250.

🔌

NodeMCU ESP8266

Has 11 usable GPIO pins — enough for 8 relay outputs + 8 switch inputs (with I2C expander if needed).

GPIO Pin Assignment (NodeMCU ESP8266)

  NodeMCU GPIO   →   Function          Notes
  ───────────────────────────────────────────────────────
  GPIO 5  (D1)   →   Relay 1 (Board1)  Safe to use
  GPIO 4  (D2)   →   Relay 2 (Board1)  Safe to use
  GPIO 0  (D3)   →   Relay 3 (Board2)  Boot-sensitive — keep HIGH at boot
  GPIO 2  (D4)   →   Relay 4 (Board2)  Boot-sensitive — keep HIGH at boot
  GPIO 14 (D5)   →   Relay 5 (Board3)  Safe to use
  GPIO 12 (D6)   →   Relay 6 (Board3)  Safe to use
  GPIO 13 (D7)   →   Relay 7 (Board4)  Safe to use
  GPIO 15 (D8)   →   Relay 8 (Board4)  Must be LOW at boot — use carefully
  GPIO 16 (D0)   →   Reserved          Deep sleep wake, avoid for relays

  INPUTS (Manual Switch detection):
  GPIO 9  (SD2)  →   Avoid — used by flash
  GPIO 10 (SD3)  →   Avoid — used by flash
  Use I2C expander (PCF8574) for more inputs if needed
🚨
Boot Pin Warning:

GPIO 0, 2, and 15 affect NodeMCU boot mode. If a relay on GPIO 0 or 2 is active LOW and the relay module pulls these pins LOW at power-on, your NodeMCU won't boot! Initialize these pins HIGH in the first line of setup(), or use opto-isolated relay modules that don't affect the GPIO during boot.

  │                                                                   │
  │  Each Relay's HIGH VOLTAGE SIDE:                                  │
  │  RELAY1: COM ←── Live | NO ──► Board1 Light via existing switch   │
  │  RELAY2: COM ←── Live | NO ──► Board1 Fan via existing switch     │
  └──────────────────────────────────────────────────────────────────┘

  SWITCHBOARD WIRING (Parallel Method — for each device):

  MAINS LIVE ──┬──── [EXISTING WALL SWITCH] ────┬──── [LIGHT] ──── NEUTRAL
               └──── [RELAY COM → NO]  ──────────┘

  NodeMCU reads switch state:
  SWITCH ──── [10kΩ to 3.3V] ──── GPIO INPUT ──── GND (via switch)

HI-LINK vs Phone Charger:

Don't use a phone charger adapter for power inside your switchboard. Use a HI-LINK HLK-PM01 — it's designed for in-wall installation, is compact, and properly rated for continuous use. It costs ₹200-350 and is significantly safer and more reliable.

Why Opto-Isolated Relay Modules?

🛡️

Opto-Isolated (Get This)

The high-voltage relay side is electrically isolated from NodeMCU using an optocoupler. A voltage spike on the AC side cannot damage your NodeMCU. Required for professional installs.

⚠️

Non-Isolated (Avoid)

Direct connection between relay coil and NodeMCU GPIO. A relay arc or spike can destroy your NodeMCU instantly. Cheaper but dangerous for mains voltage work.

Complete Professional Code

This code handles: WiFi connection, MQTT control, manual switch input with debounce, state synchronization, and OTA updates.

home_automation.ino — Full Professional Code
/*
 * Professional Room Automation System
 * NodeMCU ESP8266 + 4-Channel Relay + MQTT
 * Features: Manual switch, IoT control, state sync, OTA update
 */

#include <ESP8266WiFi.h>
#include <PubSubClient.h>      // MQTT library
#include <ArduinoOTA.h>         // Over-the-air update
#include <EEPROM.h>             // Persist state across reboots

// ── WiFi Config ──────────────────────────────────────
const char* WIFI_SSID     = "YOUR_WIFI_NAME";
const char* WIFI_PASS     = "YOUR_WIFI_PASSWORD";

// ── MQTT Config ──────────────────────────────────────
const char* MQTT_SERVER   = "broker.hivemq.com"; // Free public broker
const int   MQTT_PORT     = 1883;
const char* DEVICE_ID     = "room_controller_01";

// ── Relay Pins (Active LOW) ───────────────────────────
const int RELAY_PINS[4] = {5, 4, 14, 12}; // D1,D2,D5,D6

// ── Switch Input Pins ────────────────────────────────
const int SWITCH_PINS[4] = {13, 3, 1, 16}; // D7, RX, TX, D0
// Note: Use push buttons for best results (see Section 3, Method D)

// ── MQTT Topics ──────────────────────────────────────
const char* CMD_TOPICS[4] = {
  "home/room/relay1/set",
  "home/room/relay2/set",
  "home/room/relay3/set",
  "home/room/relay4/set"
};
const char* STATE_TOPICS[4] = {
  "home/room/relay1/state",
  "home/room/relay2/state",
  "home/room/relay3/state",
  "home/room/relay4/state"
};

// ── State Tracking ───────────────────────────────────
bool relayState[4] = {false, false, false, false};
int  lastSwitchState[4] = {HIGH, HIGH, HIGH, HIGH};
unsigned long lastDebounce[4] = {0, 0, 0, 0};
const unsigned long DEBOUNCE_MS = 50;

WiFiClient wifiClient;
PubSubClient mqtt(wifiClient);

// ── Set Relay + Publish State ─────────────────────────
void setRelay(int index, bool on) {
  relayState[index] = on;
  digitalWrite(RELAY_PINS[index], on ? LOW : HIGH); // Active LOW!

  // Publish state to MQTT (retained = true, so apps see current state)
  mqtt.publish(STATE_TOPICS[index], on ? "ON" : "OFF", true);

  // Save to EEPROM so state persists after power loss
  EEPROM.write(index, on ? 1 : 0);
  EEPROM.commit();
}

// ── MQTT Message Handler ──────────────────────────────
void onMqttMessage(char* topic, byte* payload, unsigned int len) {
  String msg = "";
  for (int i = 0; i < len; i++) msg += (char)payload[i];

  for (int i = 0; i < 4; i++) {
    if (String(topic) == String(CMD_TOPICS[i])) {
      if (msg == "ON")     setRelay(i, true);
      else if (msg == "OFF")  setRelay(i, false);
      else if (msg == "TOGGLE") setRelay(i, !relayState[i]);
    }
  }
}

// ── WiFi Connection ───────────────────────────────────
void connectWiFi() {
  WiFi.begin(WIFI_SSID, WIFI_PASS);
  while (WiFi.status() != WL_CONNECTED) {
    delay(500);
  }
}

// ── MQTT Reconnect ────────────────────────────────────
void reconnectMQTT() {
  while (!mqtt.connected()) {
    if (mqtt.connect(DEVICE_ID)) {
      for (int i = 0; i < 4; i++) {
        mqtt.subscribe(CMD_TOPICS[i]);
        // Re-publish current state on reconnect
        mqtt.publish(STATE_TOPICS[i], relayState[i] ? "ON" : "OFF", true);
      }
    } else {
      delay(5000);
    }
  }
}

// ── Setup ─────────────────────────────────────────────
void setup() {
  EEPROM.begin(16);

  // Init relays FIRST — keep them OFF during boot
  for (int i = 0; i < 4; i++) {
    pinMode(RELAY_PINS[i], OUTPUT);
    digitalWrite(RELAY_PINS[i], HIGH); // OFF (active LOW module)
  }

  // Init switch inputs with internal pull-up
  for (int i = 0; i < 4; i++) {
    pinMode(SWITCH_PINS[i], INPUT_PULLUP);
    lastSwitchState[i] = digitalRead(SWITCH_PINS[i]);
  }

  // Restore state from EEPROM
  for (int i = 0; i < 4; i++) {
    bool savedState = EEPROM.read(i) == 1;
    setRelay(i, savedState);
  }

  connectWiFi();

  // OTA Update setup
  ArduinoOTA.setHostname("room-controller");
  ArduinoOTA.begin();

  mqtt.setServer(MQTT_SERVER, MQTT_PORT);
  mqtt.setCallback(onMqttMessage);
}

// ── Main Loop ─────────────────────────────────────────
void loop() {
  ArduinoOTA.handle(); // Handle OTA updates

  if (!mqtt.connected()) reconnectMQTT();
  mqtt.loop();

  // ── Manual Switch Handling with Debounce ──
  for (int i = 0; i < 4; i++) {
    int reading = digitalRead(SWITCH_PINS[i]);

    if (reading != lastSwitchState[i]) {
      lastDebounce[i] = millis();
    }

    if ((millis() - lastDebounce[i]) > DEBOUNCE_MS) {
      if (reading == LOW) {         // Button pressed (pull-up → LOW when pressed)
        setRelay(i, !relayState[i]); // Toggle!
      }
    }

    lastSwitchState[i] = reading;
  }
}
✅
What this code handles:

WiFi auto-reconnect, MQTT auto-reconnect, debounced button presses, state persistence across power cuts (EEPROM), retained MQTT messages (apps see current state immediately), OTA firmware updates (no USB needed for future updates), and active-LOW relay logic.

Safety Rules — Non-Negotiable

☠️
230V AC is lethal. Always follow these rules:

Turn off your MCB/breaker before working on any mains wiring. Never touch live wires. Use insulated tools. Test your circuit with a non-contact voltage tester before touching anything.

🔒

Always use a junction box

House your NodeMCU and relay module in a proper IP-rated enclosure. No exposed circuits.

🌡️

Check relay ratings

Use relays rated for at least 10A/250V. Songle SRD-05VDC-SL-C is the reliable standard. Don't control motors larger than the relay can handle.

🔌

Always switch the LIVE wire

The relay/switch must interrupt the Live (phase) wire, not the neutral. Switching neutral leaves the device energized — dangerous.

🔥

Use proper wire gauge

For lights: 1mm² (standard). For fans and heaters: 1.5mm² minimum. Never use thin hookup wire for mains connections.

⚡

Add a fuse

Put a 5A or 10A fuse (as appropriate) on the Live line entering your relay module. Protects against shorts.

🛡️

Isolate AC from DC

Keep mains wiring on one side of the relay module. DC control wiring on the other side. Never mix them in the same cable run.
